The problem

Agent SQL needs runtime control, not another migration tool.

AI agents can now generate and execute SQL during ordinary workflows. A missing WHERE, broad update, unsafe function, or catalog query can alter or expose far more data than the task required.

Vindexa focuses on live agent actions against existing Postgres databases: classify the SQL, enforce policy, estimate the blast radius, hold risky writes for approval, audit the decision, and preserve an undo path when the write shape supports it.

Control path

Designed around the decisions an infrastructure team actually needs.

01

02

03

Policy and simulation

Deterministic rules run first; risky writes can be rolled back in simulation.

04

Approve, log, undo

Safe writes run, risky writes hold, reversible changes receive action ids.

Enterprise-shaped controls

AST classification

Detect reads, writes, DDL, multi-statement batches, catalog access, locking reads, unsafe functions, wrapped writes, and sensitive columns.

Policy enforcement

YAML policy returns stable machine-readable reason codes with suggested fixes.

Blast-radius simulation

Risky writes can be measured inside a timed transaction before any commit.

Approval gates

Large or contradictory writes are held with an approval id and require an operator token.

Audit visibility

Async JSONL audit logs include decisions, classifications, and queue health.

Undo protection

Reversible writes capture before/after images; unsupported shapes are blocked by default.

Status

Production-grade problem. Developer-preview implementation.

The interface is positioned for enterprise evaluation, but the repo remains a local developer preview. Before production use, add least-privilege roles, durable/tamper-resistant audit storage, hosted CI, persistent approval state, and security review.
